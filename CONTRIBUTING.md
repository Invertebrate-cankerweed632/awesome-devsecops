# Contributing to Awesome DevSecOps

Thank you for your interest in contributing! This document provides guidelines for contributing to this awesome list of DevSecOps tools, resources, and best practices.

## How to Contribute

### Adding a New Resource

1. **Fork the repository**
2. **Create a new branch** for your addition
3. **Add your resource** in the appropriate section
4. **Ensure quality** - only add resources that are:
   - High quality and well-maintained
   - Relevant to DevSecOps (security in development/operations)
   - Accessible (preferably open source or with free tier)
   - Not already listed
5. **Follow the format** described below
6. **Submit a pull request**

### Resource Format

Please use the following format for new entries:

```markdown
- [Resource Name](https://url-to-resource.com) - Brief description of what it is and why it's useful
```

**Guidelines:**
- Keep descriptions concise (1-2 sentences)
- Start with a capital letter
- End with a period
- Be objective and factual
- Avoid marketing language

### Categories

Place your resource in the most appropriate category:

- **Learning Resources** - Tutorials, courses, books, certifications
- **Code Security** - SAST, SCA, secrets detection
- **Security Testing** - DAST, fuzzing, API security
- **Container & Kubernetes Security** - Container scanning, runtime protection
- **Infrastructure Security** - IaC scanning, cloud security
- **Secrets Management** - Vault solutions, key management
- **CI/CD Security** - Pipeline security, supply chain
- **Monitoring & Response** - SIEM, incident response
- **Compliance & Policy** - Policy as code, compliance tools
- **Community** - Forums, discussions, communities

If none fit, propose a new category in your pull request.

## Types of Contributions

### 1. Adding Resources
The primary way to contribute - add valuable DevSecOps tools and resources.

### 2. Updating Information
- Fix broken links
- Update descriptions
- Add missing information

### 3. Improving Organization
- Suggest better categorization
- Propose new sections
- Improve readability

### 4. Reporting Issues
- Broken links
- Outdated information
- Inappropriate content
- Suggestions for improvement

## Pull Request Process

1. **Search existing PRs** to avoid duplicates
2. **One resource per PR** (makes review easier)
3. **Clear PR title**: "Add [Resource Name]" or "Update [Section]"
4. **Describe your addition** in the PR description:
   - What you're adding
   - Why it's valuable
   - How you discovered it

5. **Wait for review** - maintainers will review within a few days
6. **Address feedback** if requested
7. **Celebrate** when merged! üéâ

## Style Guide

### Markdown

- Use proper Markdown syntax
- Check rendering before submitting
- Keep consistent formatting

### Links

- Use HTTPS when available
- Ensure links work
- Use full URLs (not shortened links)
- Link to official sources when possible

### Descriptions

**Good examples:**
```markdown
- [Trivy](https://github.com/aquasecurity/trivy) - Comprehensive security scanner for containers, IaC, and dependencies with support for multiple formats
- [Semgrep](https://semgrep.dev/) - Fast, open-source SAST tool that finds bugs and enforces code standards across 30+ languages
```

**Bad examples:**
```markdown
- [Amazing Security Tool](https://example.com) - The best security tool ever!!!
- [My Security Blog](https://blog.com) - Check out my blog
- [Tool](https://example.com) - Security tool
```
